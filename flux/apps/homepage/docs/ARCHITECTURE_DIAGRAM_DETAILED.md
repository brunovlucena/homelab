### Common Issues:

#### Issue: Database not loading when running `make tp-frontend`

**Symptoms:**
- Frontend loads but no data from database
- API calls return empty or fail
- Works fine when intercept is stopped

**Root Cause:**
The `Dockerfile.dev` runs nginx by default, but for Telepresence intercepts with volume mounts, you need to run the Vite dev server (`npm run dev`) instead. The Vite dev server handles the API proxy correctly.

**Solution:**
The `tp-frontend` target now correctly:
1. Overrides the CMD to run `npm run dev` instead of nginx
2. Sets environment variables for proper API service resolution
3. Uses the shorter service name (`homepage-api.homepage.svc`) that works better with Telepresence DNS

**Verification:**
```bash
# Check if Vite dev server is running in the container
docker ps | grep intercept
docker logs <container-id>

# You should see Vite dev server output, not nginx
# Look for: "VITE vX.X.X  ready in XXX ms"
```

**If still not working:**
1. Check Vite proxy logs in the container - it should show which API target it's using
2. Verify API service is accessible from the container:
   ```bash
   docker exec -it <frontend-container-id> sh
   # Inside container:
   wget -O- http://homepage-api.homepage.svc:8080/api/health
   ```
3. Check if API intercept is also needed (if API is not running in cluster):
   ```bash
   make tp-api  # Run this in addition to tp-frontend
   ```

## ğŸ“Š Key Architecture Points

### React + Vite Interaction:
- **Development**: Vite serves modules directly (no bundling), enables HMR
- **Production**: Vite bundles code into optimized chunks, served by nginx
- **Proxy**: Vite proxy forwards `/api/*` to backend in dev, nginx does it in prod

### API Communication:
- **Frontend**: Always uses `/api/*` (relative URLs)
- **Dev**: Vite proxy forwards to K8s service or localhost
- **Prod**: nginx proxy forwards to K8s service
- **No code changes** needed between environments

### Database & Cache:

### 1. **Intercept Modes & Options**

#### Basic Intercept

### 2. **Docker Integration Options**

#### Docker Run (Current Approach)
```bash
# With --docker-run flag
telepresence intercept homepage-api \
  --port 8080:8080 \
  --docker-run -- \
  -v $(pwd)/src/api:/app \
  -w /app \

```

#### Docker Build (Build on the Fly)
```bash
# Build and run in one command
telepresence intercept homepage-api \
  --port 8080:8080 \
  --docker-build ./src/api \
  --docker-build-opt tag=dev \
  --docker-build-opt build-arg=ENV=dev \
  -- IMAGE
```
- Builds Docker image from context automatically
- Uses `IMAGE` as placeholder (replaced with built image ID)
- No need to build images manually

#### Docker Debug (With Debugger Support)
```bash
# For debugging with relaxed security
telepresence intercept homepage-api \
  --port 8080:8080 \
  --docker-debug ./src/api \
  -- IMAGE
```

#### Docker Daemon Mode (Recommended)
```bash
# Start daemon in Docker (no admin access needed)

# Then use intercepts normally
telepresence intercept homepage-api --port 8080:8080 --docker-run -- ...
```
**Benefits:**
- No admin/sudo access required
- No MacFUSE/WinFSP needed
- Automatic volume mounting via Telemount plugin
- Network isolated to Docker network

### 3. **Volume Mount Options**

#### Automatic Mount (Default)
```bash
telepresence intercept homepage-api \
  --port 8080:8080 \
  --mount=true \
  --docker-run -- ...
```
- Telepresence picks random mount point
- Access via `$TELEPRESENCE_ROOT` environment variable
- All remote volumes automatically mounted

#### Custom Mount Point
```bash
telepresence intercept homepage-api \
  --port 8080:8080 \
  --mount=/tmp/telepresence \
  --docker-run -- ...
```

#### Disable Mounting
```bash
telepresence intercept homepage-api \
  --port 8080:8080 \
  --mount=false \
  --docker-run -- ...
```

#### Read-Only Mounts
```bash
telepresence intercept homepage-api \
  --port 8080:8080 \
  --mount=/tmp/:ro \
  --docker-run -- ...
```

**Accessing Mounted Volumes:**
- Kubernetes secrets: `$TELEPRESENCE_ROOT/var/run/secrets/kubernetes.io`
- ConfigMaps: `$TELEPRESENCE_ROOT/etc/config`
- Any pod volume: `$TELEPRESENCE_ROOT/<original-path>`

### 4. **Environment Variable Management**

#### Export to File
```bash
# Export as Docker env file
telepresence intercept homepage-api \
  --port 8080:8080 \
  --env-file .env \
  --docker-run -- ...

# Export as JSON
telepresence intercept homepage-api \
  --port 8080:8080 \
  --env-json env.json \
  --docker-run -- ...

# Custom syntax (sh, csh, cmd, ps, compose)

  --port 8080:8080 \
  --env-file .env \
  --env-syntax=sh:export \
  --docker-run -- ...
```

#### Automatic Environment Propagation
When using `--docker-run`, environment variables are automatically propagated:
- All pod environment variables available in container
- ConfigMap/Secret env vars included
- Telepresence variables added:
  - `TELEPRESENCE_ROOT` - Volume mount root
  - `TELEPRESENCE_MOUNTS` - Colon-separated mount paths
  - `TELEPRESENCE_CONTAINER` - Container name

### 5. **HTTP Traffic Filtering**

#### Path-Based Filtering
```bash
# Exact path match
telepresence intercept homepage-api \
  --port 8080:8080 \
  --http-path-equal=/api/projects \
  --docker-run -- ...

# Path prefix (useful for API versioning)
telepresence intercept homepage-api \
  --port 8080:8080 \
  --http-path-prefix=/api/v1 \
  --docker-run -- ...

# Regex pattern
telepresence intercept homepage-api \
  --port 8080:8080 \
  --http-path-regex='^/api/(projects|skills)' \
  --docker-run -- ...
```

#### Header-Based Filtering
```bash
# Single header
telepresence intercept homepage-api \
  --port 8080:8080 \
  --http-header="X-User-ID=dev123" \
  --docker-run -- ...

# Multiple headers (AND logic)
telepresence intercept homepage-api \
  --port 8080:8080 \
  --http-header="X-User-ID=dev123" \
  --http-header="X-Environment=dev" \
  --docker-run -- ...
```

**Use Cases:**
- Multiple developers working on same service (different headers)
- Testing specific API endpoints only
- Canary testing with header-based routing

### 13. **Limitations & Considerations**

- **Volume mounts**: Must prepend `$TELEPRESENCE_ROOT` to paths
- **Network**: When using `--docker`, network is isolated to Docker
- **Port conflicts**: Ensure local ports aren't in use
- **Multiple intercepts**: Each service needs separate intercept
- **Resource usage**: Intercepts consume cluster resources (traffic-agent sidecar)
